# 工程优化路线图

本文档用于收口当前仓库下一阶段的工程优化计划。目标不是继续扩大 `0.1` 功能范围，而是在不破坏既有主链路的前提下，先降低维护成本、补齐回归防线，并把已经存在但边界未完全收口的能力整理清楚。

当前状态：

- 阶段 1 已完成，已删除页面恢复入口与对应占位 API；当前逻辑备份只承担下载职责，恢复统一走运维脚本
- 阶段 2 已完成，前端登录页、设置页控制器和 `App` 启动壳层已完成一轮结构收敛
- 阶段 3 已完成，安装、设置冲突、会话 cookie 与前端设置请求语义已补齐核心回归测试，并删除了根目录无效占位测试
- 阶段 4 已完成，周期清理任务已统一纳入后台任务管理器，审计写入已明确区分同步与后台最佳努力两种模式
- 阶段 5 已完成，关键主链路已补充运行态指标、积压统计与系统页展示，健康检查不再静默吞掉内部查询错误
- 后续阶段从“优化存储布局与媒体性能”开始继续推进

这份路线图基于当前仓库代码结构、测试覆盖、脚本和文档现状整理，适用于后续若干个迭代周期。除非明确说明，本路线图不自动提升任何功能的正式支持级别，也不改变 [`release-0.1-scope.md`](release-0.1-scope.md) 已定义的 `GA / Beta / Experimental` 边界。

## 1. 当前判断

当前仓库的主线能力已经比较完整：

- 后端已经具备安装引导、认证、图片上传/删除、运行时设置、健康检查、备份下载、审计日志等核心链路
- 前端已经具备安装向导、登录/注册/密码找回、上传页、历史图库、设置页与维护页
- 运维脚本已经覆盖 `PostgreSQL + Dragonfly + 本地存储` 的备份、恢复、PITR 与 smoke 主路径

但继续在现有结构上直接叠加功能，风险会先体现在以下几类问题上：

- 前端状态与页面控制器持续膨胀，后续改动容易牵一发动全身
- 恢复相关页面与 API 仍有残留入口，但当前正式语义已经是不支持页面恢复
- 后台异步任务与审计写入方式不统一，退出、失败可见性和运行治理不足
- 自动化测试覆盖更偏关键点和 smoke，仍缺少若干高风险状态转换的细粒度回归
- 本地存储与媒体读取在容量继续增长时会先碰到目录规模和内存占用瓶颈
- 可观测性还不够支撑长期运维，当前更接近“健康检查”而不是“运行态度量”

## 2. 优化原则

后续优化遵循以下原则：

- 优先收口边界，而不是同时扩展更多半完成能力
- 优先降低维护复杂度，而不是只追求功能数量
- 优先补高风险回归测试，而不是堆叠占位型测试
- 优先保证默认 `GA` 主链路质量，而不是把资源分散到非主线组合
- 优先使错误可见、任务可控、行为可解释，而不是依赖人工排障

## 3. 优先级

### P0

必须先做，否则会持续放大结构性债务：

- 收口“页面恢复”能力边界
- 降低前端设置页、登录页、维护页的状态与异步动作复杂度
- 补齐安装、认证、图片引用计数、设置冲突等关键回归测试

### P1

应尽快做，用于提高后端运行治理和长期可维护性：

- 统一后台任务生命周期管理
- 统一审计写入策略和错误处理语义
- 增加针对主链路的结构化指标与积压/失败可见性

### P2

在主线结构稳定后推进，用于提高容量和性能上限：

- 改进媒体读取方式，避免大文件整块读入
- 为本地存储设计分片目录布局
- 补充更贴近真实场景的浏览器与容量回归

## 4. 分阶段计划

### 阶段 1：收口产品边界

状态：已完成（2026-03-23）

目标：

- 文档、页面、接口和脚本对“恢复能力”的描述保持一致
- 不再保留误导性的半残留入口

执行项：

- 明确当前 `0.1.x` 路径下页面恢复继续保持移除状态
- 删除前端维护页中的恢复预检、安排恢复和相关提示入口
- 删除后端面向页面恢复的路由与占位实现，只保留“逻辑备份下载”和“运维脚本恢复”两条路径
- 在 PostgreSQL 文档和 README 中继续强调：后台 `backup_*.sql` 只承担下载型逻辑导出，不作为企业主恢复路径

退出标准：

- 前端不再出现“可安排恢复”的交互入口
- 后端不再暴露仅用于返回“已移除”错误的恢复接口
- 文档、页面行为、脚本说明三者口径一致

### 阶段 2：整理前端结构

状态：已完成（2026-03-23）

目标：

- 降低页面控制器复杂度
- 把重复的异步请求模式和鉴权错误处理统一起来

执行项：

- 抽象统一的页面异步动作模式，减少各页面重复编写 `spawn -> set loading -> 调服务 -> toast -> set error`
- 拆分设置页的全局控制器，把账户、安全、系统、维护、用户管理改成按 section 独立状态单元
- 整理登录页多模式表单状态，避免不同模式共享过多临时信号
- 收敛 `App` 启动阶段和页面切换中的重复分支，明确“安装态 / 未登录态 / 已登录态”三个壳层
- 统一前端鉴权失效提示和页面内错误呈现，减少同一错误在多个组件中重复处理

退出标准：

- 设置页和登录页的主要控制器明显缩小
- 重复异步动作模板被抽成共享工具
- 新增一个设置分区或一个登录子流程时，不再需要复制大段状态模板

### 阶段 3：补核心回归测试

状态：已完成（2026-03-23）

目标：

- 为最容易出问题的状态转换建立稳定回归网

优先补充的测试类型：

- 安装成功链路：初始化、管理员创建、cookie 下发、运行时设置应用
- 安装失败链路：运行时设置 apply 失败时数据库与运行态回滚
- 认证链路：refresh 轮换、会话失效、token version / session epoch 生效
- 图片链路：去重、缩略图补建、引用计数、硬删除与补偿清理
- 设置链路：`settings_version` 冲突、防并发覆盖、重启提示语义
- 运维链路：备份列表、备份下载、页面恢复移除语义、健康检查组件降级逻辑

测试策略：

- 保持关键集成测试和针对性单测并行推进
- root `tests/` 优先替换掉当前占位型样例，避免制造“看起来有测试、实际没有覆盖”的错觉
- 保持 `compose-smoke.sh` 作为主链路验收入口，但不把所有细粒度回归都推给 smoke

退出标准：

- 关键状态转换出现回归时能被单测或集成测试直接拦截
- 占位型测试显著减少
- smoke 脚本聚焦端到端验收，不再承担过多细节验证职责

### 阶段 4：统一后台任务与审计治理

状态：已完成（2026-03-23）

目标：

- 让后台周期任务和异步审计写入变得可控、可观测、可解释

执行项：

- 为过期图片清理、吊销 token 清理、存储补偿清理引入统一的后台任务管理器
- 统一任务启动、日志字段、错误计数和停止语义
- 统一审计写入策略，明确哪些路径同步写入、哪些路径允许异步写入
- 减少随处散落的裸 `tokio::spawn`，避免后续继续复制无监管后台任务

退出标准：

- 后台任务有统一入口和生命周期边界
- 审计日志写入失败不再只靠零散日志定位
- 新增后台任务时不需要复制无限循环模板

### 阶段 5：补运行态指标与可观测性

状态：已完成（2026-03-23）

目标：

- 从“知道服务活着”提升到“知道主链路是否在退化”

执行项：

- 为清理任务、备份任务、审计写入、认证刷新、图片处理增加结构化指标
- 暴露可反映积压与失败趋势的运行态信息，例如：
  - 存储清理任务积压量
  - 备份耗时与失败次数
  - 审计写入失败次数
  - refresh 失败次数
  - 图片处理耗时分布
- 继续保留当前健康检查的人类可读输出，但避免把内部查询错误直接静默吞掉

退出标准：

- 可以从指标看出主链路是否开始退化
- 排障不再只依赖用户反馈或手工翻日志

### 阶段 6：优化存储布局与媒体性能

目标：

- 提高图片容量上限，降低大文件读取开销

执行项：

- 将媒体响应从整块读入内存逐步改为流式返回
- 补充条件请求支持，例如 `ETag` / `304` 复用
- 将本地物理存储从单层文件平铺改为按 hash 前缀分片目录
- 设计兼容迁移方案，使已有部署可逐步从旧布局过渡到新布局

注意：

- 这一阶段改动面较大，应放在结构和回归测试相对稳定之后推进
- 在迁移方案未准备好前，不应直接改写现有文件布局

退出标准：

- 单目录文件数量不再线性膨胀
- 大图访问不再默认整块读入
- 新旧布局迁移有明确兼容策略和回退方案

## 5. 非目标

本路线图不包含以下目标：

- 扩大 `0.1` 的正式支持矩阵
- 恢复所有数据库后端的页面内恢复能力
- 在没有回归测试和运维语义之前引入更多新存储后端
- 为了“看起来功能更多”而保留误导性的占位入口

## 6. 建议执行顺序

建议按以下顺序推进：

1. 阶段 1：收口页面恢复边界
2. 阶段 2：整理前端状态与控制器
3. 阶段 3：补关键回归测试
4. 阶段 4：统一后台任务与审计治理
5. 阶段 5：补运行态指标
6. 阶段 6：优化存储布局与媒体性能

原因：

- 前三步主要解决“语义不一致、结构复杂、回归网不够”这三类最容易持续放大的问题
- 后三步更偏运行治理和容量优化，适合在结构先稳定之后推进

## 7. 文档治理要求

后续如果路线图发生调整，应同步检查以下文档是否需要更新：

- [`README.md`](../README.md)
- [`release-0.1-scope.md`](release-0.1-scope.md)
- [`postgresql.md`](postgresql.md)
- 与发布门禁、运维演练和页面行为相关的 runbook

原则上：

- 能力被移除时，要同步删除对应入口和文档描述
- 能力未进入正式支持级别时，不应在首页文档中伪装成默认能力
- 路线图完成某阶段后，应把对应结论沉淀回正式文档，而不是只停留在这份计划中
